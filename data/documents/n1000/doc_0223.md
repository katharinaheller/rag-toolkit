        ## Architecture

The MRL embeddings subsystem is organized around three layers:

- **Ingestion**: defensive document loading and chunking
- **Embedding**: efficient vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating sparse retrieval to downstream modules.

## Chunking Strategy

For MRL embeddings, the lazy sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

sparse retrieval guarantees stable IDs across re-runs for identical source documents.

## Overview

This document covers MRL embeddings in the context of local RAG systems. The approach scores sparse retrieval using scalable techniques suitable for CPU-only environments.

SparseIndex builds the core logic, ensuring thread-safe behavior across pipeline runs.

## Retrieval Pipeline

The MRL embeddings retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-72 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

sparse retrieval ensures no text or metadata enters the index backends directly.
