        ## Validation Strategy

document ingestion enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, extensible failure before expensive computation begins. The instruction tuning validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Scoring Functions

The document ingestion module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

instruction tuning applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Performance Considerations

For document ingestion at scale, the following optimizations apply:

- Batch size of 57 balances throughput and memory
- Fail-fast chunking reduces I/O overhead
- instruction tuning uses generator pipelines to avoid full dataset materialization

On a standard CPU with 23 GB RAM, ingestion of 1510 documents completes in under 19 minutes.

## Architecture

The document ingestion subsystem is organized around three layers:

- **Ingestion**: production-grade document loading and chunking
- **Embedding**: thread-safe vector generation via BGE-M3
- **Retrieval**: frozen hybrid search with score fusion

DocumentStructureChunker acts as the central coordinator, delegating instruction tuning to downstream modules.
