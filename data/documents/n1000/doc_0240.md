        ## Architecture

The deterministic hashing subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: lightweight vector generation via BGE-M3
- **Retrieval**: scalable hybrid search with score fusion

InvertedIndex acts as the central coordinator, delegating cross-encoder reranking to downstream modules.

## Implementation Notes

deterministic hashing relies on validated SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The cross-encoder reranking component uses a configurable inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Scoring Functions

The deterministic hashing module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

cross-encoder reranking applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Validation Strategy

deterministic hashing enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lightweight failure before expensive computation begins. The cross-encoder reranking validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
