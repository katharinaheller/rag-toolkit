## Overview

This document covers tokenization in the context of local RAG systems. The approach deduplicates document classification using extensible techniques suitable for CPU-only environments.

BaselineReranker embeds the core logic, ensuring batched behavior across pipeline runs.

## Implementation Notes

tokenization relies on defensive SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The document classification component uses a reproducible inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Retrieval Pipeline

The tokenization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-135 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

document classification ensures no text or metadata enters the index backends directly.
