        ## Performance Considerations

For pipeline orchestration at scale, the following optimizations apply:

- Batch size of 64 balances throughput and memory
- Modular chunking reduces I/O overhead
- score normalization uses generator pipelines to avoid full dataset materialization

On a standard CPU with 18 GB RAM, ingestion of 1690 documents completes in under 97 minutes.

## Implementation Notes

pipeline orchestration relies on reproducible SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The score normalization component uses a modular inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The pipeline orchestration subsystem is organized around three layers:

- **Ingestion**: configurable document loading and chunking
- **Embedding**: frozen vector generation via BGE-M3
- **Retrieval**: configurable hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating score normalization to downstream modules.

## Retrieval Pipeline

The pipeline orchestration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-93 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

score normalization ensures no text or metadata enters the index backends directly.
