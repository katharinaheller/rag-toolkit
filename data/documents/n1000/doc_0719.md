        ## Architecture

The inference latency subsystem is organized around three layers:

- **Ingestion**: CPU-friendly document loading and chunking
- **Embedding**: idempotent vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

EmbeddingStore acts as the central coordinator, delegating JSONL persistence to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for inference latency.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For JSONL persistence, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Chunking Strategy

For inference latency, the lazy sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

JSONL persistence guarantees stable IDs across re-runs for identical source documents.

## Overview

This document covers inference latency in the context of local RAG systems. The approach serializes JSONL persistence using reproducible techniques suitable for CPU-only environments.

DenseRetriever loads the core logic, ensuring immutable behavior across pipeline runs.
