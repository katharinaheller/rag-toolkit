        ## Architecture

The context window management subsystem is organized around three layers:

- **Ingestion**: incremental document loading and chunking
- **Embedding**: incremental vector generation via BGE-M3
- **Retrieval**: immutable hybrid search with score fusion

DenseRetriever acts as the central coordinator, delegating gemma to downstream modules.

## Scoring Functions

The context window management module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

gemma applies sublinear TF scaling for term frequency, controlled via `sublinear_tf=True`.

## Validation Strategy

context window management enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, scalable failure before expensive computation begins. The gemma validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
