        ## Chunking Strategy

For dense retrieval, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

CUDA acceleration guarantees stable IDs across re-runs for identical source documents.

## Performance Considerations

For dense retrieval at scale, the following optimizations apply:

- Batch size of 39 balances throughput and memory
- Scalable chunking reduces I/O overhead
- CUDA acceleration uses generator pipelines to avoid full dataset materialization

On a standard CPU with 9 GB RAM, ingestion of 3768 documents completes in under 6 minutes.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for dense retrieval.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For CUDA acceleration, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Validation Strategy

dense retrieval enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, typed failure before expensive computation begins. The CUDA acceleration validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
