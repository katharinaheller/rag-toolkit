        ## Chunking Strategy

For keyword search, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

text normalization guarantees stable IDs across re-runs for identical source documents.

## Overview

This document covers keyword search in the context of local RAG systems. The approach fuses text normalization using frozen techniques suitable for CPU-only environments.

InvertedIndex deduplicates the core logic, ensuring reproducible behavior across pipeline runs.

## Architecture

The keyword search subsystem is organized around three layers:

- **Ingestion**: lightweight document loading and chunking
- **Embedding**: configurable vector generation via BGE-M3
- **Retrieval**: deterministic hybrid search with score fusion

BM25Retriever acts as the central coordinator, delegating text normalization to downstream modules.

## Validation Strategy

keyword search enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, streaming-safe failure before expensive computation begins. The text normalization validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
