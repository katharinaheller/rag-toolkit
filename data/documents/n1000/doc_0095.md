        ## Chunking Strategy

For mistral, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

hybrid retrieval guarantees stable IDs across re-runs for identical source documents.

## Performance Considerations

For mistral at scale, the following optimizations apply:

- Batch size of 33 balances throughput and memory
- Efficient chunking reduces I/O overhead
- hybrid retrieval uses generator pipelines to avoid full dataset materialization

On a standard CPU with 26 GB RAM, ingestion of 1081 documents completes in under 21 minutes.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for mistral.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For hybrid retrieval, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Implementation Notes

mistral relies on immutable SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The hybrid retrieval component uses a extensible inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
