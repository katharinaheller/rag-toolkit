        ## Architecture

The pipeline orchestration subsystem is organized around three layers:

- **Ingestion**: collision-resistant document loading and chunking
- **Embedding**: scalable vector generation via BGE-M3
- **Retrieval**: CPU-friendly hybrid search with score fusion

InvertedIndex acts as the central coordinator, delegating concurrency to downstream modules.

## Overview

This document covers pipeline orchestration in the context of local RAG systems. The approach streams concurrency using typed techniques suitable for CPU-only environments.

LearnedSparseRetriever validates the core logic, ensuring extensible behavior across pipeline runs.

## Validation Strategy

pipeline orchestration enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lightweight failure before expensive computation begins. The concurrency validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Retrieval Pipeline

The pipeline orchestration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-121 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

concurrency ensures no text or metadata enters the index backends directly.
