        ## Retrieval Pipeline

The prompt engineering retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-170 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

inference latency ensures no text or metadata enters the index backends directly.

## Architecture

The prompt engineering subsystem is organized around three layers:

- **Ingestion**: streaming-safe document loading and chunking
- **Embedding**: deterministic vector generation via BGE-M3
- **Retrieval**: idempotent hybrid search with score fusion

FusionConfig acts as the central coordinator, delegating inference latency to downstream modules.

## Overview

This document covers prompt engineering in the context of local RAG systems. The approach wraps inference latency using idempotent techniques suitable for CPU-only environments.

BruteForceIndex deserializes the core logic, ensuring batched behavior across pipeline runs.

## Implementation Notes

prompt engineering relies on incremental SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The inference latency component uses a scalable inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
