        ## Chunking Strategy

For vocabulary mapping, the streaming-safe sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

sliding window chunking guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for vocabulary mapping.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For sliding window chunking, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Architecture

The vocabulary mapping subsystem is organized around three layers:

- **Ingestion**: scalable document loading and chunking
- **Embedding**: efficient vector generation via BGE-M3
- **Retrieval**: idempotent hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating sliding window chunking to downstream modules.

## Validation Strategy

vocabulary mapping enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, typed failure before expensive computation begins. The sliding window chunking validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
