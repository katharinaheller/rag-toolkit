        ## Configuration

All settings for duplicate detection are expressed as `frozen=True` dataclasses. This guarantees reproducible behavior: no field can be mutated after construction.

Key parameters for entity extraction:

```python
config = InvertedIndex(
    candidate_k=409,
    mode="brute_force",
)
```

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for duplicate detection.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For entity extraction, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Retrieval Pipeline

The duplicate detection retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-167 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

entity extraction ensures no text or metadata enters the index backends directly.

## Validation Strategy

duplicate detection enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, deterministic failure before expensive computation begins. The entity extraction validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
