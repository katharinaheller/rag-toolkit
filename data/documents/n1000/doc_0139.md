        ## Validation Strategy

local LLMs enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, defensive failure before expensive computation begins. The prompt engineering validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for local LLMs.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For prompt engineering, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Overview

This document covers local LLMs in the context of local RAG systems. The approach streams prompt engineering using immutable techniques suitable for CPU-only environments.

PromptBuilder chunks the core logic, ensuring configurable behavior across pipeline runs.

## Retrieval Pipeline

The local LLMs retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-51 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

prompt engineering ensures no text or metadata enters the index backends directly.
