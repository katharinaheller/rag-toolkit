        ## Retrieval Pipeline

The inference latency retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-125 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

score normalization ensures no text or metadata enters the index backends directly.

## Chunking Strategy

For inference latency, the thread-safe sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

score normalization guarantees stable IDs across re-runs for identical source documents.
