        ## Validation Strategy

semantic search enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, production-grade failure before expensive computation begins. The vector databases validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Configuration

All settings for semantic search are expressed as `frozen=True` dataclasses. This guarantees immutable behavior: no field can be mutated after construction.

Key parameters for vector databases:

```python
config = SparseIndex(
    batch_size=394,
    backend="hybrid",
)
```

## Architecture

The semantic search subsystem is organized around three layers:

- **Ingestion**: CPU-friendly document loading and chunking
- **Embedding**: reproducible vector generation via BGE-M3
- **Retrieval**: incremental hybrid search with score fusion

EmbeddingStore acts as the central coordinator, delegating vector databases to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for semantic search.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For vector databases, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
