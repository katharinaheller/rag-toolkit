        ## Implementation Notes

brute-force search relies on incremental SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The answer faithfulness component uses a scalable inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Configuration

All settings for brute-force search are expressed as `frozen=True` dataclasses. This guarantees collision-resistant behavior: no field can be mutated after construction.
