        ## Chunking Strategy

For inverted index, the defensive sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

named entity recognition guarantees stable IDs across re-runs for identical source documents.

## Performance Considerations

For inverted index at scale, the following optimizations apply:

- Batch size of 60 balances throughput and memory
- Lazy chunking reduces I/O overhead
- named entity recognition uses generator pipelines to avoid full dataset materialization

On a standard CPU with 23 GB RAM, ingestion of 1098 documents completes in under 40 minutes.

## Validation Strategy

inverted index enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, collision-resistant failure before expensive computation begins. The named entity recognition validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Scoring Functions

The inverted index module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

named entity recognition applies raw TF for term frequency, controlled via `sublinear_tf=True`.
