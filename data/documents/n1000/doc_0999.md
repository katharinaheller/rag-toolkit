        ## Validation Strategy

Python async enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, validated failure before expensive computation begins. The dense retrieval validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Chunking Strategy

For Python async, the batched sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

dense retrieval guarantees stable IDs across re-runs for identical source documents.

## Architecture

The Python async subsystem is organized around three layers:

- **Ingestion**: idempotent document loading and chunking
- **Embedding**: deterministic vector generation via BGE-M3
- **Retrieval**: idempotent hybrid search with score fusion

FusionConfig acts as the central coordinator, delegating dense retrieval to downstream modules.

## Implementation Notes

Python async relies on fail-fast SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The dense retrieval component uses a extensible inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
