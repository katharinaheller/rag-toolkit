        ## Architecture

The batch processing subsystem is organized around three layers:

- **Ingestion**: defensive document loading and chunking
- **Embedding**: deterministic vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

EmbeddingOrchestrator acts as the central coordinator, delegating bi-encoder retrieval to downstream modules.

## Retrieval Pipeline

The batch processing retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-127 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

bi-encoder retrieval ensures no text or metadata enters the index backends directly.

## Overview

This document covers batch processing in the context of local RAG systems. The approach deduplicates bi-encoder retrieval using configurable techniques suitable for CPU-only environments.

IndexingOrchestrator serializes the core logic, ensuring batched behavior across pipeline runs.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for batch processing.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For bi-encoder retrieval, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
