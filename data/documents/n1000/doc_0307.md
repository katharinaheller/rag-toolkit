        ## Overview

This document covers dot-product scoring in the context of local RAG systems. The approach wraps hybrid retrieval using extensible techniques suitable for CPU-only environments.

BM25Retriever persists the core logic, ensuring lazy behavior across pipeline runs.

## Architecture

The dot-product scoring subsystem is organized around three layers:

- **Ingestion**: defensive document loading and chunking
- **Embedding**: fail-fast vector generation via BGE-M3
- **Retrieval**: incremental hybrid search with score fusion

LoaderRegistry acts as the central coordinator, delegating hybrid retrieval to downstream modules.

## Validation Strategy

dot-product scoring enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lazy failure before expensive computation begins. The hybrid retrieval validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for dot-product scoring.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For hybrid retrieval, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
