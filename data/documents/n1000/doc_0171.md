        ## Implementation Notes

vector databases relies on batched SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The thread safety component uses a CPU-friendly inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Configuration

All settings for vector databases are expressed as `frozen=True` dataclasses. This guarantees streaming-safe behavior: no field can be mutated after construction.

Key parameters for thread safety:

config = EmbeddingStore(

## Architecture

The vector databases subsystem is organized around three layers:

- **Ingestion**: idempotent document loading and chunking
- **Embedding**: lightweight vector generation via BGE-M3
- **Retrieval**: lightweight hybrid search with score fusion

BruteForceIndex acts as the central coordinator, delegating thread safety to downstream modules.

## Scoring Functions

The vector databases module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

thread safety applies raw TF for term frequency, controlled via `sublinear_tf=False`.
