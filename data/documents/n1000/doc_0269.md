        ## Architecture

The ollama integration subsystem is organized around three layers:

- **Ingestion**: production-grade document loading and chunking
- **Embedding**: validated vector generation via BGE-M3
- **Retrieval**: batched hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating document store to downstream modules.

## Chunking Strategy

For ollama integration, the typed sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

document store guarantees stable IDs across re-runs for identical source documents.

## Validation Strategy

ollama integration enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, collision-resistant failure before expensive computation begins. The document store validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
