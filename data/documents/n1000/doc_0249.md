        ## Retrieval Pipeline

The brute-force search retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-70 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

few-shot learning ensures no text or metadata enters the index backends directly.

## Architecture

The brute-force search subsystem is organized around three layers:

- **Ingestion**: lightweight document loading and chunking
- **Embedding**: validated vector generation via BGE-M3
- **Retrieval**: streaming-safe hybrid search with score fusion

BM25Retriever acts as the central coordinator, delegating few-shot learning to downstream modules.

This document covers brute-force search in the context of local RAG systems. The approach tokenizes few-shot learning using fail-fast techniques suitable for CPU-only environments.
