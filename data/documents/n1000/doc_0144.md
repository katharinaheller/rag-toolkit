        ## Implementation Notes

ColBERT relies on incremental SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The topic modeling component uses a CPU-friendly inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Validation Strategy

ColBERT enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, thread-safe failure before expensive computation begins. The topic modeling validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The ColBERT subsystem is organized around three layers:

- **Ingestion**: reproducible document loading and chunking
- **Embedding**: immutable vector generation via BGE-M3
- **Retrieval**: streaming-safe hybrid search with score fusion

DenseRetriever acts as the central coordinator, delegating topic modeling to downstream modules.

## Retrieval Pipeline

The ColBERT retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-72 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

topic modeling ensures no text or metadata enters the index backends directly.
