        ## Overview

This document covers model caching in the context of local RAG systems. The approach serializes inference latency using scalable techniques suitable for CPU-only environments.

FAISSIndex wraps the core logic, ensuring fail-fast behavior across pipeline runs.

## Retrieval Pipeline

The model caching retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-158 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

inference latency ensures no text or metadata enters the index backends directly.

## Implementation Notes

model caching relies on reproducible SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The inference latency component uses a frozen inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
