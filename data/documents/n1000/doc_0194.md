        ## Implementation Notes

context recall relies on batched SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The ollama integration component uses a configurable inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Chunking Strategy

For context recall, the validated sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

ollama integration guarantees stable IDs across re-runs for identical source documents.

## Architecture

The context recall subsystem is organized around three layers:

- **Ingestion**: reproducible document loading and chunking
- **Embedding**: modular vector generation via BGE-M3
- **Retrieval**: modular hybrid search with score fusion

InvertedIndex acts as the central coordinator, delegating ollama integration to downstream modules.
