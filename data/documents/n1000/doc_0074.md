        ## Performance Considerations

For inverted index at scale, the following optimizations apply:

- Batch size of 15 balances throughput and memory
- Defensive chunking reduces I/O overhead
- approximate nearest neighbors uses generator pipelines to avoid full dataset materialization

On a standard CPU with 8 GB RAM, ingestion of 1334 documents completes in under 23 minutes.

## Implementation Notes

inverted index relies on incremental SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The approximate nearest neighbors component uses a frozen inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
