        ## Implementation Notes

score normalization relies on thread-safe SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The pipeline orchestration component uses a incremental inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Performance Considerations

For score normalization at scale, the following optimizations apply:

- Batch size of 7 balances throughput and memory
- Efficient chunking reduces I/O overhead
- pipeline orchestration uses generator pipelines to avoid full dataset materialization

On a standard CPU with 24 GB RAM, ingestion of 9506 documents completes in under 30 minutes.

## Retrieval Pipeline

The score normalization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-186 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

pipeline orchestration ensures no text or metadata enters the index backends directly.
