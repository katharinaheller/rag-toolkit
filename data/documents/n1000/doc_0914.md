        ## Chunking Strategy

For dense retrieval, the batched sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

document store guarantees stable IDs across re-runs for identical source documents.

## Configuration

All settings for dense retrieval are expressed as `frozen=True` dataclasses. This guarantees CPU-friendly behavior: no field can be mutated after construction.

## Architecture

The dense retrieval subsystem is organized around three layers:

- **Ingestion**: configurable document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: batched hybrid search with score fusion

DenseRetriever acts as the central coordinator, delegating document store to downstream modules.
