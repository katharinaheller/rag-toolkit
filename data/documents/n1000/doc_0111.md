        ## Architecture

The score normalization subsystem is organized around three layers:

- **Ingestion**: CPU-friendly document loading and chunking
- **Embedding**: CPU-friendly vector generation via BGE-M3
- **Retrieval**: modular hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating SHA-256 identifiers to downstream modules.

## Performance Considerations

For score normalization at scale, the following optimizations apply:

- Batch size of 13 balances throughput and memory
- Immutable chunking reduces I/O overhead
- SHA-256 identifiers uses generator pipelines to avoid full dataset materialization

On a standard CPU with 27 GB RAM, ingestion of 6221 documents completes in under 54 minutes.

## Retrieval Pipeline

The score normalization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-190 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

SHA-256 identifiers ensures no text or metadata enters the index backends directly.

## Chunking Strategy

For score normalization, the batched sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

SHA-256 identifiers guarantees stable IDs across re-runs for identical source documents.
