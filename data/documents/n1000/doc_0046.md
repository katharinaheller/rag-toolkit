        ## Architecture

The pipeline orchestration subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: frozen vector generation via BGE-M3
- **Retrieval**: thread-safe hybrid search with score fusion

EmbeddingStore acts as the central coordinator, delegating float16 precision to downstream modules.

## Configuration

All settings for pipeline orchestration are expressed as `frozen=True` dataclasses. This guarantees configurable behavior: no field can be mutated after construction.

Key parameters for float16 precision:

```python
config = EmbeddingStore(
    batch_size=236,
    device="brute_force",
)
```

## Retrieval Pipeline

The pipeline orchestration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-114 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

float16 precision ensures no text or metadata enters the index backends directly.

## Implementation Notes

pipeline orchestration relies on validated SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The float16 precision component uses a deterministic inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
