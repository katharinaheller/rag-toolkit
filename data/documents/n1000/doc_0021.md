        ## Retrieval Pipeline

The BM25 ranking retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-148 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

BGE-M3 embeddings ensures no text or metadata enters the index backends directly.

## Validation Strategy

BM25 ranking enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lazy failure before expensive computation begins. The BGE-M3 embeddings validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The BM25 ranking subsystem is organized around three layers:

- **Ingestion**: extensible document loading and chunking
- **Embedding**: deterministic vector generation via BGE-M3
- **Retrieval**: validated hybrid search with score fusion

FusionConfig acts as the central coordinator, delegating BGE-M3 embeddings to downstream modules.
