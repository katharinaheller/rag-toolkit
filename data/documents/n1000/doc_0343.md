        ## Chunking Strategy

For VRAM management, the immutable sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

entity extraction guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for VRAM management.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For entity extraction, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Performance Considerations

For VRAM management at scale, the following optimizations apply:

- Batch size of 16 balances throughput and memory
- Immutable chunking reduces I/O overhead
- entity extraction uses generator pipelines to avoid full dataset materialization

On a standard CPU with 6 GB RAM, ingestion of 1413 documents completes in under 51 minutes.
