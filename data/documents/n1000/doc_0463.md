        ## Architecture

The dense retrieval subsystem is organized around three layers:

- **Ingestion**: batched document loading and chunking
- **Embedding**: lightweight vector generation via BGE-M3
- **Retrieval**: lightweight hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating tokenization to downstream modules.

## Implementation Notes

dense retrieval relies on extensible SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The tokenization component uses a validated inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for dense retrieval.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For tokenization, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Validation Strategy

dense retrieval enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, scalable failure before expensive computation begins. The tokenization validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
