        ## Retrieval Pipeline

The gemma retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-166 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

inference latency ensures no text or metadata enters the index backends directly.

## Overview

This document covers gemma in the context of local RAG systems. The approach serializes inference latency using immutable techniques suitable for CPU-only environments.

IngestionPipeline configures the core logic, ensuring batched behavior across pipeline runs.

## Implementation Notes

gemma relies on CPU-friendly SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The inference latency component uses a deterministic inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
