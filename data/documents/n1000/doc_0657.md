        ## Architecture

The prompt engineering subsystem is organized around three layers:

- **Ingestion**: batched document loading and chunking
- **Embedding**: typed vector generation via BGE-M3
- **Retrieval**: idempotent hybrid search with score fusion

EmbeddingOrchestrator acts as the central coordinator, delegating approximate nearest neighbors to downstream modules.

## Implementation Notes

prompt engineering relies on frozen SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The approximate nearest neighbors component uses a lightweight inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Configuration

All settings for prompt engineering are expressed as `frozen=True` dataclasses. This guarantees idempotent behavior: no field can be mutated after construction.

Key parameters for approximate nearest neighbors:

```python

    top_k=285,
```
