        ## Validation Strategy

cross-encoder reranking enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, defensive failure before expensive computation begins. The inference latency validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The cross-encoder reranking subsystem is organized around three layers:

- **Ingestion**: collision-resistant document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: scalable hybrid search with score fusion

JSONLStore acts as the central coordinator, delegating inference latency to downstream modules.

## Retrieval Pipeline

The cross-encoder reranking retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-126 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

inference latency ensures no text or metadata enters the index backends directly.
