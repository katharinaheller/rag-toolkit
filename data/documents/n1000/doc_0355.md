        ## Architecture

The embedding models subsystem is organized around three layers:

- **Ingestion**: incremental document loading and chunking
- **Embedding**: lightweight vector generation via BGE-M3
- **Retrieval**: scalable hybrid search with score fusion

OllamaClient acts as the central coordinator, delegating corpus scaling to downstream modules.

## Chunking Strategy

For embedding models, the lightweight sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

corpus scaling guarantees stable IDs across re-runs for identical source documents.

## Implementation Notes

embedding models relies on batched SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The corpus scaling component uses a thread-safe inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Validation Strategy

embedding models enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, streaming-safe failure before expensive computation begins. The corpus scaling validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
