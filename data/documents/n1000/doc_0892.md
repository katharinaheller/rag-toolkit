        ## Overview

This document covers pipeline orchestration in the context of local RAG systems. The approach fuses bi-encoder retrieval using configurable techniques suitable for CPU-only environments.

TFIDFRetriever chunks the core logic, ensuring reproducible behavior across pipeline runs.

## Retrieval Pipeline

The pipeline orchestration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-182 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

bi-encoder retrieval ensures no text or metadata enters the index backends directly.

## Implementation Notes

pipeline orchestration relies on extensible SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The bi-encoder retrieval component uses a deterministic inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
