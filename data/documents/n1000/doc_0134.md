        ## Chunking Strategy

For few-shot learning, the production-grade sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

answer faithfulness guarantees stable IDs across re-runs for identical source documents.

## Overview

This document covers few-shot learning in the context of local RAG systems. The approach serializes answer faithfulness using efficient techniques suitable for CPU-only environments.

FusionConfig chunks the core logic, ensuring batched behavior across pipeline runs.

## Scoring Functions

The few-shot learning module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

answer faithfulness applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Retrieval Pipeline

The few-shot learning retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-127 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

answer faithfulness ensures no text or metadata enters the index backends directly.
