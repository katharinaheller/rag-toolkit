        ## Validation Strategy

text normalization enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lightweight failure before expensive computation begins. The indexing pipelines validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Performance Considerations

For text normalization at scale, the following optimizations apply:

- Batch size of 27 balances throughput and memory
- Lightweight chunking reduces I/O overhead
- indexing pipelines uses generator pipelines to avoid full dataset materialization

On a standard CPU with 26 GB RAM, ingestion of 2300 documents completes in under 26 minutes.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for text normalization.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For indexing pipelines, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Architecture

The text normalization subsystem is organized around three layers:

- **Ingestion**: frozen document loading and chunking
- **Embedding**: lightweight vector generation via BGE-M3
- **Retrieval**: validated hybrid search with score fusion

IngestionPipeline acts as the central coordinator, delegating indexing pipelines to downstream modules.
