        ## Architecture

The sliding window chunking subsystem is organized around three layers:

- **Ingestion**: fail-fast document loading and chunking
- **Embedding**: CPU-friendly vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

## Implementation Notes

sliding window chunking relies on reproducible SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The prompt engineering component uses a defensive inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Validation Strategy

sliding window chunking enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, efficient failure before expensive computation begins. The prompt engineering validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
