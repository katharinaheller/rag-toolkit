        ## Retrieval Pipeline

The score normalization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-178 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

sparse retrieval ensures no text or metadata enters the index backends directly.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for score normalization.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For sparse retrieval, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Performance Considerations

For score normalization at scale, the following optimizations apply:

- Batch size of 60 balances throughput and memory
- Configurable chunking reduces I/O overhead
- sparse retrieval uses generator pipelines to avoid full dataset materialization

On a standard CPU with 32 GB RAM, ingestion of 582 documents completes in under 72 minutes.

## Architecture

The score normalization subsystem is organized around three layers:

- **Ingestion**: streaming-safe document loading and chunking
- **Embedding**: CPU-friendly vector generation via BGE-M3
- **Retrieval**: modular hybrid search with score fusion

HybridRetriever acts as the central coordinator, delegating sparse retrieval to downstream modules.
