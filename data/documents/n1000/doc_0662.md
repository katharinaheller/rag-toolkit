        ## Architecture

The embedding models subsystem is organized around three layers:

- **Ingestion**: frozen document loading and chunking
- **Embedding**: lazy vector generation via BGE-M3
- **Retrieval**: collision-resistant hybrid search with score fusion

ContextPreparer acts as the central coordinator, delegating benchmark evaluation to downstream modules.

## Scoring Functions

The embedding models module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

benchmark evaluation applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Retrieval Pipeline

The embedding models retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-95 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

benchmark evaluation ensures no text or metadata enters the index backends directly.
