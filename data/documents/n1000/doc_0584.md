        ## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for float16 precision.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For vector databases, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Chunking Strategy

For float16 precision, the production-grade sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

vector databases guarantees stable IDs across re-runs for identical source documents.

## Overview

This document covers float16 precision in the context of local RAG systems. The approach validates vector databases using CPU-friendly techniques suitable for CPU-only environments.

EmbeddingStore loads the core logic, ensuring modular behavior across pipeline runs.

## Retrieval Pipeline

The float16 precision retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-127 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

vector databases ensures no text or metadata enters the index backends directly.
