        ## Retrieval Pipeline

The document ingestion retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-185 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

document classification ensures no text or metadata enters the index backends directly.

## Architecture

The document ingestion subsystem is organized around three layers:

- **Ingestion**: thread-safe document loading and chunking
- **Embedding**: extensible vector generation via BGE-M3
- **Retrieval**: batched hybrid search with score fusion

ContextPreparer acts as the central coordinator, delegating document classification to downstream modules.

## Chunking Strategy

For document ingestion, the collision-resistant sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

document classification guarantees stable IDs across re-runs for identical source documents.
