        ## Implementation Notes

BGE-M3 embeddings relies on lightweight SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The GPU memory component uses a configurable inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for BGE-M3 embeddings.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For GPU memory, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Architecture

The BGE-M3 embeddings subsystem is organized around three layers:

- **Ingestion**: extensible document loading and chunking
- **Embedding**: efficient vector generation via BGE-M3
- **Retrieval**: reproducible hybrid search with score fusion

OllamaClient acts as the central coordinator, delegating GPU memory to downstream modules.
