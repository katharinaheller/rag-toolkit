        ## Implementation Notes

tokenization relies on batched SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The knowledge graphs component uses a batched inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Scoring Functions

The tokenization module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

knowledge graphs applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Retrieval Pipeline

The tokenization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-192 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

knowledge graphs ensures no text or metadata enters the index backends directly.
