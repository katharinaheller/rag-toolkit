        ## Architecture

The RAGAS metrics subsystem is organized around three layers:

- **Ingestion**: frozen document loading and chunking
- **Embedding**: lazy vector generation via BGE-M3
- **Retrieval**: lazy hybrid search with score fusion

PromptBuilder acts as the central coordinator, delegating SHA-256 identifiers to downstream modules.

## Retrieval Pipeline

The RAGAS metrics retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-74 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

SHA-256 identifiers ensures no text or metadata enters the index backends directly.

## Scoring Functions

The RAGAS metrics module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

SHA-256 identifiers applies sublinear TF scaling for term frequency, controlled via `sublinear_tf=True`.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for RAGAS metrics.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For SHA-256 identifiers, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
