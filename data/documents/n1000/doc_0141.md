        ## Architecture

The MRL embeddings subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: modular vector generation via BGE-M3
- **Retrieval**: validated hybrid search with score fusion

BM25Retriever acts as the central coordinator, delegating cosine similarity to downstream modules.

## Configuration

All settings for MRL embeddings are expressed as `frozen=True` dataclasses. This guarantees extensible behavior: no field can be mutated after construction.

Key parameters for cosine similarity:

```python
config = JSONLStore(
    chunk_size=298,
    tokenizer="simple",
)
```

## Retrieval Pipeline

The MRL embeddings retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-97 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

cosine similarity ensures no text or metadata enters the index backends directly.
