        ## Validation Strategy

named entity recognition enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, validated failure before expensive computation begins. The CPU inference validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Retrieval Pipeline

The named entity recognition retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-176 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

CPU inference ensures no text or metadata enters the index backends directly.

## Implementation Notes

named entity recognition relies on modular SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The CPU inference component uses a incremental inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The named entity recognition subsystem is organized around three layers:

- **Ingestion**: lightweight document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

OllamaClient acts as the central coordinator, delegating CPU inference to downstream modules.
