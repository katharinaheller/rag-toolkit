        ## Retrieval Pipeline

The llama retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-79 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

citation generation ensures no text or metadata enters the index backends directly.

## Validation Strategy

llama enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, deterministic failure before expensive computation begins. The citation generation validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
