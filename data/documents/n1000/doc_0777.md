        ## Retrieval Pipeline

The retrieval-augmented generation retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-104 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

subword tokenization ensures no text or metadata enters the index backends directly.

## Chunking Strategy

For retrieval-augmented generation, the streaming-safe sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

subword tokenization guarantees stable IDs across re-runs for identical source documents.

## Configuration

All settings for retrieval-augmented generation are expressed as `frozen=True` dataclasses. This guarantees lightweight behavior: no field can be mutated after construction.

Key parameters for subword tokenization:

## Implementation Notes

retrieval-augmented generation relies on streaming-safe SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The subword tokenization component uses a configurable inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
