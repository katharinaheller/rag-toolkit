        ## Retrieval Pipeline

The pipeline orchestration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-174 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

JSONL persistence ensures no text or metadata enters the index backends directly.

## Overview

This document covers pipeline orchestration in the context of local RAG systems. The approach chunks JSONL persistence using incremental techniques suitable for CPU-only environments.

BaselineReranker validates the core logic, ensuring immutable behavior across pipeline runs.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for pipeline orchestration.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For JSONL persistence, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
