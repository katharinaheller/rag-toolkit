        ## Implementation Notes

named entity recognition relies on immutable SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The knowledge graphs component uses a production-grade inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Performance Considerations

For named entity recognition at scale, the following optimizations apply:

- Batch size of 22 balances throughput and memory
- Incremental chunking reduces I/O overhead
- knowledge graphs uses generator pipelines to avoid full dataset materialization

On a standard CPU with 17 GB RAM, ingestion of 6723 documents completes in under 56 minutes.

## Retrieval Pipeline

The named entity recognition retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-146 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

knowledge graphs ensures no text or metadata enters the index backends directly.

## Architecture

The named entity recognition subsystem is organized around three layers:

- **Ingestion**: frozen document loading and chunking
- **Embedding**: scalable vector generation via BGE-M3
- **Retrieval**: collision-resistant hybrid search with score fusion

ContextPreparer acts as the central coordinator, delegating knowledge graphs to downstream modules.
