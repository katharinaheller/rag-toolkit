        ## Architecture

The score normalization subsystem is organized around three layers:

- **Ingestion**: deterministic document loading and chunking
- **Embedding**: extensible vector generation via BGE-M3
- **Retrieval**: incremental hybrid search with score fusion

ChunkStore acts as the central coordinator, delegating multimodal retrieval to downstream modules.

## Retrieval Pipeline

The score normalization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-161 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

multimodal retrieval ensures no text or metadata enters the index backends directly.

## Scoring Functions

The score normalization module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

multimodal retrieval applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Chunking Strategy

For score normalization, the production-grade sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

multimodal retrieval guarantees stable IDs across re-runs for identical source documents.
