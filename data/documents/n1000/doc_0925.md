        ## Implementation Notes

Python async relies on lightweight SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The BGE-M3 embeddings component uses a idempotent inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The Python async subsystem is organized around three layers:

- **Ingestion**: defensive document loading and chunking
- **Embedding**: production-grade vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating BGE-M3 embeddings to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for Python async.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For BGE-M3 embeddings, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
