## Validation Strategy

CPU inference enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, batched failure before expensive computation begins. The text normalization validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The CPU inference subsystem is organized around three layers:

- **Ingestion**: lightweight document loading and chunking
- **Embedding**: validated vector generation via BGE-M3
- **Retrieval**: defensive hybrid search with score fusion

TFIDFRetriever acts as the central coordinator, delegating text normalization to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for CPU inference.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For text normalization, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
