## Implementation Notes

RAG evaluation relies on thread-safe SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The type safety component uses a fail-fast inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Performance Considerations

For RAG evaluation at scale, the following optimizations apply:

- Batch size of 63 balances throughput and memory
- Incremental chunking reduces I/O overhead
- type safety uses generator pipelines to avoid full dataset materialization

## Architecture

The RAG evaluation subsystem is organized around three layers:

- **Ingestion**: configurable document loading and chunking
- **Embedding**: extensible vector generation via BGE-M3
- **Retrieval**: scalable hybrid search with score fusion

DocumentStore acts as the central coordinator, delegating type safety to downstream modules.
