        ## Chunking Strategy

For metadata extraction, the reproducible sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

entity extraction guarantees stable IDs across re-runs for identical source documents.

## Architecture

The metadata extraction subsystem is organized around three layers:

- **Ingestion**: typed document loading and chunking
- **Embedding**: lazy vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

RetrievalOrchestrator acts as the central coordinator, delegating entity extraction to downstream modules.

## Validation Strategy

metadata extraction enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, efficient failure before expensive computation begins. The entity extraction validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Implementation Notes

metadata extraction relies on efficient SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The entity extraction component uses a streaming-safe inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
