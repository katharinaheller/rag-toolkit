        ## Implementation Notes

CPU inference relies on idempotent SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The context window management component uses a lightweight inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The CPU inference subsystem is organized around three layers:

- **Ingestion**: configurable document loading and chunking
- **Embedding**: configurable vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

BruteForceIndex acts as the central coordinator, delegating context window management to downstream modules.

## Validation Strategy

CPU inference enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, deterministic failure before expensive computation begins. The context window management validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
