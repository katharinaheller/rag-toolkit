        ## Chunking Strategy

For GPU memory, the validated sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

dense retrieval guarantees stable IDs across re-runs for identical source documents.

## Configuration

All settings for GPU memory are expressed as `frozen=True` dataclasses. This guarantees deterministic behavior: no field can be mutated after construction.

Key parameters for dense retrieval:

```python
config = JSONLStore(
    chunk_size=470,
    device="simple",
)
```

## Scoring Functions

The GPU memory module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

dense retrieval applies raw TF for term frequency, controlled via `sublinear_tf=True`.

## Implementation Notes

GPU memory relies on batched SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The dense retrieval component uses a validated inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
