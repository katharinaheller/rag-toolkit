        ## Architecture

The BGE-M3 embeddings subsystem is organized around three layers:

- **Ingestion**: batched document loading and chunking
- **Embedding**: collision-resistant vector generation via BGE-M3
- **Retrieval**: modular hybrid search with score fusion

BruteForceIndex acts as the central coordinator, delegating inverted index to downstream modules.

## Configuration

All settings for BGE-M3 embeddings are expressed as `frozen=True` dataclasses. This guarantees frozen behavior: no field can be mutated after construction.

Key parameters for inverted index:

```python
config = HybridRetriever(
    candidate_k=53,
    backend="cpu",
)
```

## Retrieval Pipeline

The BGE-M3 embeddings retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-167 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

inverted index ensures no text or metadata enters the index backends directly.

## Implementation Notes

BGE-M3 embeddings relies on collision-resistant SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The inverted index component uses a thread-safe inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
