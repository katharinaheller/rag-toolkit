        ## Chunking Strategy

For reproducibility, the idempotent sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

dataclass configuration guarantees stable IDs across re-runs for identical source documents.

## Architecture

The reproducibility subsystem is organized around three layers:

- **Ingestion**: frozen document loading and chunking
- **Embedding**: production-grade vector generation via BGE-M3
- **Retrieval**: configurable hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating dataclass configuration to downstream modules.

## Retrieval Pipeline

The reproducibility retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-141 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

dataclass configuration ensures no text or metadata enters the index backends directly.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for reproducibility.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For dataclass configuration, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
