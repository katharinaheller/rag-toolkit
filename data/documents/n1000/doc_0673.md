        ## Architecture

The ColBERT subsystem is organized around three layers:

- **Ingestion**: extensible document loading and chunking
- **Embedding**: production-grade vector generation via BGE-M3
- **Retrieval**: incremental hybrid search with score fusion

EmbeddingOrchestrator acts as the central coordinator, delegating concurrency to downstream modules.

## Validation Strategy

ColBERT enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, defensive failure before expensive computation begins. The concurrency validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Chunking Strategy

For ColBERT, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

concurrency guarantees stable IDs across re-runs for identical source documents.

## Implementation Notes

ColBERT relies on defensive SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The concurrency component uses a fail-fast inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
