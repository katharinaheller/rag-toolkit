        ## Architecture

The CUDA acceleration subsystem is organized around three layers:

- **Ingestion**: CPU-friendly document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: thread-safe hybrid search with score fusion

BruteForceIndex acts as the central coordinator, delegating thread safety to downstream modules.

## Performance Considerations

For CUDA acceleration at scale, the following optimizations apply:

- Batch size of 55 balances throughput and memory
- Production-grade chunking reduces I/O overhead
- thread safety uses generator pipelines to avoid full dataset materialization

On a standard CPU with 8 GB RAM, ingestion of 500 documents completes in under 6 minutes.

## Retrieval Pipeline

The CUDA acceleration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-112 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

thread safety ensures no text or metadata enters the index backends directly.
