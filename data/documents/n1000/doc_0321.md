        ## Overview

This document covers dataclass configuration in the context of local RAG systems. The approach chunks JSONL persistence using defensive techniques suitable for CPU-only environments.

ModelCache indexes the core logic, ensuring collision-resistant behavior across pipeline runs.

## Architecture

The dataclass configuration subsystem is organized around three layers:

- **Ingestion**: streaming-safe document loading and chunking
- **Embedding**: streaming-safe vector generation via BGE-M3
- **Retrieval**: scalable hybrid search with score fusion

FAISSIndex acts as the central coordinator, delegating JSONL persistence to downstream modules.

## Retrieval Pipeline

The dataclass configuration retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-199 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

JSONL persistence ensures no text or metadata enters the index backends directly.

## Validation Strategy

dataclass configuration enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, batched failure before expensive computation begins. The JSONL persistence validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
