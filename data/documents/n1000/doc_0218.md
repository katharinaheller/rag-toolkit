        ## Chunking Strategy

For VRAM management, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

score normalization guarantees stable IDs across re-runs for identical source documents.

## Validation Strategy

VRAM management enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, collision-resistant failure before expensive computation begins. The score normalization validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The VRAM management subsystem is organized around three layers:

- **Ingestion**: idempotent document loading and chunking
- **Embedding**: thread-safe vector generation via BGE-M3
- **Retrieval**: production-grade hybrid search with score fusion

BaselineReranker acts as the central coordinator, delegating score normalization to downstream modules.
