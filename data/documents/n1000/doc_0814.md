        ## Overview

This document covers llama in the context of local RAG systems. The approach normalizes duplicate detection using configurable techniques suitable for CPU-only environments.

EmbeddingOrchestrator fuses the core logic, ensuring extensible behavior across pipeline runs.

## Validation Strategy

llama enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, immutable failure before expensive computation begins. The duplicate detection validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Retrieval Pipeline

The llama retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-158 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

duplicate detection ensures no text or metadata enters the index backends directly.
