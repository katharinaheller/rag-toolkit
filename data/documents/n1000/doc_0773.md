        ## Validation Strategy

chunk store enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, extensible failure before expensive computation begins. The inverted index validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Chunking Strategy

For chunk store, the incremental sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

inverted index guarantees stable IDs across re-runs for identical source documents.

## Overview

This document covers chunk store in the context of local RAG systems. The approach persists inverted index using collision-resistant techniques suitable for CPU-only environments.

OllamaClient processes the core logic, ensuring reproducible behavior across pipeline runs.

## Retrieval Pipeline

The chunk store retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-122 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

inverted index ensures no text or metadata enters the index backends directly.
