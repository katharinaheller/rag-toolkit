        ## Performance Considerations

For vocabulary mapping at scale, the following optimizations apply:

- Batch size of 45 balances throughput and memory
- Cpu-friendly chunking reduces I/O overhead
- FAISS indexing uses generator pipelines to avoid full dataset materialization

On a standard CPU with 20 GB RAM, ingestion of 4483 documents completes in under 41 minutes.

## Retrieval Pipeline

The vocabulary mapping retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-188 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

FAISS indexing ensures no text or metadata enters the index backends directly.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for vocabulary mapping.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For FAISS indexing, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Validation Strategy

vocabulary mapping enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, extensible failure before expensive computation begins. The FAISS indexing validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
