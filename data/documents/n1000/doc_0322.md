        ## Overview

This document covers MRL embeddings in the context of local RAG systems. The approach configures keyword search using validated techniques suitable for CPU-only environments.

IngestionPipeline builds the core logic, ensuring immutable behavior across pipeline runs.

## Architecture

The MRL embeddings subsystem is organized around three layers:

- **Ingestion**: idempotent document loading and chunking
- **Embedding**: CPU-friendly vector generation via BGE-M3
- **Retrieval**: thread-safe hybrid search with score fusion

FAISSIndex acts as the central coordinator, delegating keyword search to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for MRL embeddings.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For keyword search, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Implementation Notes

MRL embeddings relies on production-grade SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The keyword search component uses a reproducible inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
