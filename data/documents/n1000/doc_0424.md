        ## Architecture

The data deduplication subsystem is organized around three layers:

- **Ingestion**: extensible document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: lightweight hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating inverted index to downstream modules.

## Configuration

All settings for data deduplication are expressed as `frozen=True` dataclasses. This guarantees defensive behavior: no field can be mutated after construction.

Key parameters for inverted index:

```python
config = LearnedSparseRetriever(
    chunk_size=17,
    device="brute_force",
)
```

## Performance Considerations

For data deduplication at scale, the following optimizations apply:

- Batch size of 28 balances throughput and memory
- Scalable chunking reduces I/O overhead
- inverted index uses generator pipelines to avoid full dataset materialization

On a standard CPU with 14 GB RAM, ingestion of 2434 documents completes in under 53 minutes.
