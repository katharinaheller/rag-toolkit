        ## Overview

This document covers keyword search in the context of local RAG systems. The approach serializes indexing pipelines using efficient techniques suitable for CPU-only environments.

HybridRetriever persists the core logic, ensuring reproducible behavior across pipeline runs.

## Retrieval Pipeline

The keyword search retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-115 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

indexing pipelines ensures no text or metadata enters the index backends directly.

## Implementation Notes

keyword search relies on configurable SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The indexing pipelines component uses a defensive inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Chunking Strategy

For keyword search, the validated sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

indexing pipelines guarantees stable IDs across re-runs for identical source documents.
