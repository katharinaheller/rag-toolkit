        ## Architecture

The knowledge graphs subsystem is organized around three layers:

- **Ingestion**: batched document loading and chunking
- **Embedding**: typed vector generation via BGE-M3
- **Retrieval**: frozen hybrid search with score fusion

EmbeddingOrchestrator acts as the central coordinator, delegating approximate nearest neighbors to downstream modules.

## Configuration

All settings for knowledge graphs are expressed as `frozen=True` dataclasses. This guarantees configurable behavior: no field can be mutated after construction.

Key parameters for approximate nearest neighbors:

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for knowledge graphs.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For approximate nearest neighbors, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Retrieval Pipeline

The knowledge graphs retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-197 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

approximate nearest neighbors ensures no text or metadata enters the index backends directly.
