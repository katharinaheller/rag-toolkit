## Retrieval Pipeline

The approximate nearest neighbors retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-148 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

reciprocal rank fusion ensures no text or metadata enters the index backends directly.

## Implementation Notes

approximate nearest neighbors relies on efficient SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The reciprocal rank fusion component uses a fail-fast inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The approximate nearest neighbors subsystem is organized around three layers:

- **Ingestion**: defensive document loading and chunking
- **Embedding**: streaming-safe vector generation via BGE-M3
- **Retrieval**: frozen hybrid search with score fusion

IngestionPipeline acts as the central coordinator, delegating reciprocal rank fusion to downstream modules.
