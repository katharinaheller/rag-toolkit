        ## Architecture

The chunking strategies subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: modular vector generation via BGE-M3
- **Retrieval**: incremental hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating type safety to downstream modules.

## Chunking Strategy

For chunking strategies, the idempotent sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

type safety guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for chunking strategies.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For type safety, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
