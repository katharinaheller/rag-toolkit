        ## Chunking Strategy

For chunk store, the lightweight sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

keyword search guarantees stable IDs across re-runs for identical source documents.

## Architecture

The chunk store subsystem is organized around three layers:

- **Ingestion**: validated document loading and chunking
- **Embedding**: modular vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

RetrievalOrchestrator acts as the central coordinator, delegating keyword search to downstream modules.

## Performance Considerations

For chunk store at scale, the following optimizations apply:

- Batch size of 64 balances throughput and memory
- Cpu-friendly chunking reduces I/O overhead
- keyword search uses generator pipelines to avoid full dataset materialization
