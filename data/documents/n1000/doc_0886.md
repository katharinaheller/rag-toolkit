        ## Retrieval Pipeline

The vocabulary mapping retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-106 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

chain-of-thought prompting ensures no text or metadata enters the index backends directly.

## Architecture

The vocabulary mapping subsystem is organized around three layers:

- **Ingestion**: scalable document loading and chunking
- **Embedding**: lazy vector generation via BGE-M3
- **Retrieval**: defensive hybrid search with score fusion

LearnedSparseRetriever acts as the central coordinator, delegating chain-of-thought prompting to downstream modules.

## Validation Strategy

vocabulary mapping enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, validated failure before expensive computation begins. The chain-of-thought prompting validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for vocabulary mapping.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For chain-of-thought prompting, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
