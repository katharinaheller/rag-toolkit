## Retrieval Pipeline

The indexing pipelines retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-149 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

RAGAS metrics ensures no text or metadata enters the index backends directly.

## Chunking Strategy

For indexing pipelines, the idempotent sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

RAGAS metrics guarantees stable IDs across re-runs for identical source documents.

## Architecture

The indexing pipelines subsystem is organized around three layers:

- **Ingestion**: batched document loading and chunking
- **Embedding**: lazy vector generation via BGE-M3
- **Retrieval**: thread-safe hybrid search with score fusion

OllamaClient acts as the central coordinator, delegating RAGAS metrics to downstream modules.
