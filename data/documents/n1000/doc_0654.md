        ## Retrieval Pipeline

The BM25 ranking retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-171 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

bfloat16 precision ensures no text or metadata enters the index backends directly.

## Overview

This document covers BM25 ranking in the context of local RAG systems. The approach persists bfloat16 precision using idempotent techniques suitable for CPU-only environments.

SparseIndex persists the core logic, ensuring immutable behavior across pipeline runs.

## Implementation Notes

BM25 ranking relies on configurable SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The bfloat16 precision component uses a frozen inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
