        ## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for few-shot learning.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For LRU eviction, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Implementation Notes

few-shot learning relies on efficient SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The LRU eviction component uses a streaming-safe inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Chunking Strategy

For few-shot learning, the extensible sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

LRU eviction guarantees stable IDs across re-runs for identical source documents.

## Architecture

The few-shot learning subsystem is organized around three layers:

- **Ingestion**: scalable document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

TFIDFRetriever acts as the central coordinator, delegating LRU eviction to downstream modules.
