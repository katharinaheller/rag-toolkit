        ## Validation Strategy

brute-force search enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, incremental failure before expensive computation begins. The ablation studies validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Retrieval Pipeline

The brute-force search retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-70 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

ablation studies ensures no text or metadata enters the index backends directly.

## Architecture

The brute-force search subsystem is organized around three layers:

- **Ingestion**: lightweight document loading and chunking
- **Embedding**: fail-fast vector generation via BGE-M3
- **Retrieval**: reproducible hybrid search with score fusion

RetrievalOrchestrator acts as the central coordinator, delegating ablation studies to downstream modules.
