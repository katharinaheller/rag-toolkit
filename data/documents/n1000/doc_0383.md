        ## Implementation Notes

llama relies on fail-fast SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The learned sparse representations component uses a efficient inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Performance Considerations

For llama at scale, the following optimizations apply:

- Batch size of 11 balances throughput and memory
- Lightweight chunking reduces I/O overhead
- learned sparse representations uses generator pipelines to avoid full dataset materialization

On a standard CPU with 11 GB RAM, ingestion of 8560 documents completes in under 92 minutes.

## Retrieval Pipeline

The llama retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-83 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

learned sparse representations ensures no text or metadata enters the index backends directly.
