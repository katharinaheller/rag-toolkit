        ## Retrieval Pipeline

The noise removal retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-167 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

chunk store ensures no text or metadata enters the index backends directly.

## Overview

This document covers noise removal in the context of local RAG systems. The approach streams chunk store using extensible techniques suitable for CPU-only environments.

InvertedIndex fuses the core logic, ensuring streaming-safe behavior across pipeline runs.

## Architecture

The noise removal subsystem is organized around three layers:

- **Ingestion**: collision-resistant document loading and chunking
- **Embedding**: thread-safe vector generation via BGE-M3
- **Retrieval**: collision-resistant hybrid search with score fusion

ContextPreparer acts as the central coordinator, delegating chunk store to downstream modules.
