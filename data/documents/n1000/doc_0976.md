        ## Chunking Strategy

For multimodal retrieval, the extensible sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

vector databases guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for multimodal retrieval.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For vector databases, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Validation Strategy

multimodal retrieval enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, extensible failure before expensive computation begins. The vector databases validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The multimodal retrieval subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: incremental vector generation via BGE-M3
- **Retrieval**: modular hybrid search with score fusion

IngestionPipeline acts as the central coordinator, delegating vector databases to downstream modules.
