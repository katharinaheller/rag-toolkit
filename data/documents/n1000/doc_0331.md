        ## Validation Strategy

indexing pipelines enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, batched failure before expensive computation begins. The few-shot learning validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Retrieval Pipeline

The indexing pipelines retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-197 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

few-shot learning ensures no text or metadata enters the index backends directly.

## Architecture

The indexing pipelines subsystem is organized around three layers:

- **Ingestion**: reproducible document loading and chunking
- **Embedding**: fail-fast vector generation via BGE-M3
- **Retrieval**: lightweight hybrid search with score fusion

HybridRetriever acts as the central coordinator, delegating few-shot learning to downstream modules.
