        ## Retrieval Pipeline

The throughput optimization retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-53 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

hybrid retrieval ensures no text or metadata enters the index backends directly.

## Implementation Notes

throughput optimization relies on incremental SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The hybrid retrieval component uses a validated inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The throughput optimization subsystem is organized around three layers:

- **Ingestion**: streaming-safe document loading and chunking
- **Embedding**: idempotent vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating hybrid retrieval to downstream modules.
