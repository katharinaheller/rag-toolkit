        ## Architecture

The dependency injection subsystem is organized around three layers:

- **Ingestion**: defensive document loading and chunking
- **Embedding**: frozen vector generation via BGE-M3
- **Retrieval**: batched hybrid search with score fusion

RetrievalOrchestrator acts as the central coordinator, delegating RAG evaluation to downstream modules.

## Implementation Notes

dependency injection relies on collision-resistant SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The RAG evaluation component uses a reproducible inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Configuration

All settings for dependency injection are expressed as `frozen=True` dataclasses. This guarantees extensible behavior: no field can be mutated after construction.

Key parameters for RAG evaluation:

```python
config = DocumentStructureChunker(
    top_k=82,
    tokenizer="simple",
)
```
