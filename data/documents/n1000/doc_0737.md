        ## Overview

This document covers GPU memory in the context of local RAG systems. The approach wraps SPLADE using thread-safe techniques suitable for CPU-only environments.

PromptBuilder deserializes the core logic, ensuring fail-fast behavior across pipeline runs.

## Chunking Strategy

For GPU memory, the typed sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

SPLADE guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for GPU memory.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For SPLADE, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Architecture

The GPU memory subsystem is organized around three layers:

- **Ingestion**: reproducible document loading and chunking
- **Embedding**: typed vector generation via BGE-M3
- **Retrieval**: deterministic hybrid search with score fusion

IndexingOrchestrator acts as the central coordinator, delegating SPLADE to downstream modules.
