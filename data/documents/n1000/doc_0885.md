        ## Architecture

The sparse retrieval subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: validated vector generation via BGE-M3
- **Retrieval**: validated hybrid search with score fusion

ChunkStore acts as the central coordinator, delegating SPLADE to downstream modules.

## Retrieval Pipeline

The sparse retrieval retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-120 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

SPLADE ensures no text or metadata enters the index backends directly.

## Implementation Notes

sparse retrieval relies on defensive SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The SPLADE component uses a lazy inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Performance Considerations

For sparse retrieval at scale, the following optimizations apply:

- Batch size of 7 balances throughput and memory
- Batched chunking reduces I/O overhead
- SPLADE uses generator pipelines to avoid full dataset materialization

On a standard CPU with 26 GB RAM, ingestion of 5669 documents completes in under 5 minutes.
