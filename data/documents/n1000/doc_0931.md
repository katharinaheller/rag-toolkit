        ## Architecture

The full-text search subsystem is organized around three layers:

- **Ingestion**: validated document loading and chunking
- **Embedding**: validated vector generation via BGE-M3
- **Retrieval**: defensive hybrid search with score fusion

BM25Retriever acts as the central coordinator, delegating cross-encoder reranking to downstream modules.

## Performance Considerations

For full-text search at scale, the following optimizations apply:

- Batch size of 28 balances throughput and memory
- Fail-fast chunking reduces I/O overhead
- cross-encoder reranking uses generator pipelines to avoid full dataset materialization

On a standard CPU with 23 GB RAM, ingestion of 8138 documents completes in under 19 minutes.

## Chunking Strategy

For full-text search, the frozen sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

cross-encoder reranking guarantees stable IDs across re-runs for identical source documents.
