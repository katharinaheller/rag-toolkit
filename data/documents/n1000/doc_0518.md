        ## Validation Strategy

vector databases enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, immutable failure before expensive computation begins. The noise removal validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Scoring Functions

The vector databases module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

noise removal applies raw TF for term frequency, controlled via `sublinear_tf=False`.

## Architecture

The vector databases subsystem is organized around three layers:

- **Ingestion**: configurable document loading and chunking
- **Embedding**: defensive vector generation via BGE-M3
- **Retrieval**: streaming-safe hybrid search with score fusion

InvertedIndex acts as the central coordinator, delegating noise removal to downstream modules.

## Chunking Strategy

For vector databases, the lightweight sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

noise removal guarantees stable IDs across re-runs for identical source documents.
