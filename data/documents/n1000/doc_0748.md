        ## Retrieval Pipeline

The reproducibility retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-185 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

concurrency ensures no text or metadata enters the index backends directly.

## Validation Strategy

reproducibility enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lazy failure before expensive computation begins. The concurrency validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
