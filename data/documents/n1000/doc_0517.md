        ## Architecture

The full-text search subsystem is organized around three layers:

- **Ingestion**: frozen document loading and chunking
- **Embedding**: extensible vector generation via BGE-M3
- **Retrieval**: validated hybrid search with score fusion

DocumentStructureChunker acts as the central coordinator, delegating subword tokenization to downstream modules.

## Validation Strategy

full-text search enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, deterministic failure before expensive computation begins. The subword tokenization validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for full-text search.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For subword tokenization, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
