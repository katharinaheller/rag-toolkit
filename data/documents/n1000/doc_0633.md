        ## Chunking Strategy

For sparse retrieval, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

GPU memory guarantees stable IDs across re-runs for identical source documents.

## Architecture

The sparse retrieval subsystem is organized around three layers:

- **Ingestion**: CPU-friendly document loading and chunking
- **Embedding**: immutable vector generation via BGE-M3
- **Retrieval**: idempotent hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating GPU memory to downstream modules.

## Scoring Functions

The sparse retrieval module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

GPU memory applies sublinear TF scaling for term frequency, controlled via `sublinear_tf=False`.
