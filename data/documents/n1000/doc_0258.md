        ## Architecture

The embedding models subsystem is organized around three layers:

- **Ingestion**: modular document loading and chunking
- **Embedding**: efficient vector generation via BGE-M3
- **Retrieval**: incremental hybrid search with score fusion

BM25Retriever acts as the central coordinator, delegating hybrid retrieval to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for embedding models.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For hybrid retrieval, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Performance Considerations

For embedding models at scale, the following optimizations apply:

- Batch size of 23 balances throughput and memory
- Cpu-friendly chunking reduces I/O overhead
- hybrid retrieval uses generator pipelines to avoid full dataset materialization

On a standard CPU with 11 GB RAM, ingestion of 4755 documents completes in under 57 minutes.

## Scoring Functions

The embedding models module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

hybrid retrieval applies sublinear TF scaling for term frequency, controlled via `sublinear_tf=True`.
