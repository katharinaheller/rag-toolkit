        ## Validation Strategy

knowledge graphs enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, lightweight failure before expensive computation begins. The batch processing validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Architecture

The knowledge graphs subsystem is organized around three layers:

- **Ingestion**: typed document loading and chunking
- **Embedding**: thread-safe vector generation via BGE-M3
- **Retrieval**: collision-resistant hybrid search with score fusion

IngestionPipeline acts as the central coordinator, delegating batch processing to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for knowledge graphs.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For batch processing, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Performance Considerations

For knowledge graphs at scale, the following optimizations apply:

- Batch size of 9 balances throughput and memory
- Collision-resistant chunking reduces I/O overhead
- batch processing uses generator pipelines to avoid full dataset materialization

On a standard CPU with 25 GB RAM, ingestion of 4943 documents completes in under 40 minutes.
