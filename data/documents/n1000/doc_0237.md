        ## Retrieval Pipeline

The factory pattern retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-84 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

brute-force search ensures no text or metadata enters the index backends directly.

## Architecture

The factory pattern subsystem is organized around three layers:

- **Ingestion**: efficient document loading and chunking
- **Embedding**: thread-safe vector generation via BGE-M3
- **Retrieval**: defensive hybrid search with score fusion

ContextPreparer acts as the central coordinator, delegating brute-force search to downstream modules.

## Validation Strategy

factory pattern enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, extensible failure before expensive computation begins. The brute-force search validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.
