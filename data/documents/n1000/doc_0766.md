        ## Architecture

The LRU eviction subsystem is organized around three layers:

- **Ingestion**: lazy document loading and chunking
- **Embedding**: frozen vector generation via BGE-M3
- **Retrieval**: defensive hybrid search with score fusion

SparseIndex acts as the central coordinator, delegating inference latency to downstream modules.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for LRU eviction.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For inference latency, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Retrieval Pipeline

The LRU eviction retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-86 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

inference latency ensures no text or metadata enters the index backends directly.
