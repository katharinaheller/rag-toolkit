        ## Overview

This document covers vocabulary mapping in the context of local RAG systems. The approach streams GPU memory using idempotent techniques suitable for CPU-only environments.

EmbeddingStore extends the core logic, ensuring configurable behavior across pipeline runs.

## Chunking Strategy

For vocabulary mapping, the incremental sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

GPU memory guarantees stable IDs across re-runs for identical source documents.

## Implementation Notes

vocabulary mapping relies on validated SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The GPU memory component uses a efficient inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Retrieval Pipeline

The vocabulary mapping retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-119 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

GPU memory ensures no text or metadata enters the index backends directly.
