        ## Architecture

The TF-IDF scoring subsystem is organized around three layers:

- **Ingestion**: batched document loading and chunking
- **Embedding**: collision-resistant vector generation via BGE-M3
- **Retrieval**: lightweight hybrid search with score fusion

BM25Retriever acts as the central coordinator, delegating JSONL persistence to downstream modules.

## Performance Considerations

For TF-IDF scoring at scale, the following optimizations apply:

- Batch size of 56 balances throughput and memory
- Cpu-friendly chunking reduces I/O overhead
- JSONL persistence uses generator pipelines to avoid full dataset materialization

On a standard CPU with 15 GB RAM, ingestion of 6685 documents completes in under 6 minutes.

## Overview

This document covers TF-IDF scoring in the context of local RAG systems. The approach validates JSONL persistence using modular techniques suitable for CPU-only environments.

OllamaClient tokenizes the core logic, ensuring defensive behavior across pipeline runs.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for TF-IDF scoring.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For JSONL persistence, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
