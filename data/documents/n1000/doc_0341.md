        ## Architecture

The local LLMs subsystem is organized around three layers:

- **Ingestion**: reproducible document loading and chunking
- **Embedding**: thread-safe vector generation via BGE-M3
- **Retrieval**: extensible hybrid search with score fusion

BaselineReranker acts as the central coordinator, delegating concurrency to downstream modules.

## Validation Strategy

local LLMs enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, streaming-safe failure before expensive computation begins. The concurrency validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Retrieval Pipeline

The local LLMs retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-188 from each leg
3. **Score fusion**: weighted sum with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

concurrency ensures no text or metadata enters the index backends directly.
