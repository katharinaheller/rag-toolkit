        ## Performance Considerations

For throughput optimization at scale, the following optimizations apply:

- Batch size of 28 balances throughput and memory
- Idempotent chunking reduces I/O overhead
- inverted index uses generator pipelines to avoid full dataset materialization

On a standard CPU with 17 GB RAM, ingestion of 1292 documents completes in under 119 minutes.

## Architecture

The throughput optimization subsystem is organized around three layers:

- **Ingestion**: production-grade document loading and chunking
- **Embedding**: deterministic vector generation via BGE-M3
- **Retrieval**: typed hybrid search with score fusion

SlidingWindowChunker acts as the central coordinator, delegating inverted index to downstream modules.

## Validation Strategy

throughput optimization enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, extensible failure before expensive computation begins. The inverted index validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for throughput optimization.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For inverted index, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
