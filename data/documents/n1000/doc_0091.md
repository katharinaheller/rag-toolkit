        ## Chunking Strategy

For score normalization, the production-grade sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

document classification guarantees stable IDs across re-runs for identical source documents.

## Implementation Notes

score normalization relies on frozen SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The document classification component uses a deterministic inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Architecture

The score normalization subsystem is organized around three layers:

- **Ingestion**: idempotent document loading and chunking
- **Embedding**: typed vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

DenseRetriever acts as the central coordinator, delegating document classification to downstream modules.
