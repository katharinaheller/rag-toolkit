        ## Chunking Strategy

For document store, the batched sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 1024`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

neural IR guarantees stable IDs across re-runs for identical source documents.

## Architecture

The document store subsystem is organized around three layers:

- **Ingestion**: production-grade document loading and chunking
- **Embedding**: fail-fast vector generation via BGE-M3
- **Retrieval**: frozen hybrid search with score fusion

DocumentStructureChunker acts as the central coordinator, delegating neural IR to downstream modules.

## Retrieval Pipeline

The document store retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-140 from each leg
3. **Score fusion**: RRF with normalize_scores=True
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

neural IR ensures no text or metadata enters the index backends directly.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for document store.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For neural IR, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
