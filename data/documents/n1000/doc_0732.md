        ## Performance Considerations

For tokenization at scale, the following optimizations apply:

- Batch size of 7 balances throughput and memory
- Batched chunking reduces I/O overhead
- dataclass configuration uses generator pipelines to avoid full dataset materialization

On a standard CPU with 15 GB RAM, ingestion of 1664 documents completes in under 22 minutes.

## Overview

This document covers tokenization in the context of local RAG systems. The approach implements dataclass configuration using validated techniques suitable for CPU-only environments.

EmbeddingOrchestrator implements the core logic, ensuring collision-resistant behavior across pipeline runs.

## Chunking Strategy

For tokenization, the configurable sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

dataclass configuration guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for tokenization.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For dataclass configuration, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
