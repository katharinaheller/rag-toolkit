        ## Scoring Functions

The gemma module provides pure, stateless scoring functions:

**BM25** — Robertson-Sparck Jones smoothed IDF:
```
IDF(t) = log((N - df(t) + 0.5) / (df(t) + 0.5) + 1)
```

**TF-IDF** — smoothed variant:
```
IDF(t) = log((N + 1) / (df(t) + 1)) + 1
```

chunking strategies applies sublinear TF scaling for term frequency, controlled via `sublinear_tf=True`.

## Validation Strategy

gemma enforces a three-layer uniqueness contract:

```
DOCUMENT_ID → chunk_id → embedding_vector_id
```

Any collision triggers an immediate, frozen failure before expensive computation begins. The chunking strategies validator reports every duplicate with its source file path for fast diagnosis.

Policy: **no auto-deduplication**, **no silent skipping**.

## Chunking Strategy

For gemma, the configurable sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 64`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

chunking strategies guarantees stable IDs across re-runs for identical source documents.

## Architecture

The gemma subsystem is organized around three layers:

- **Ingestion**: production-grade document loading and chunking
- **Embedding**: lazy vector generation via BGE-M3
- **Retrieval**: fail-fast hybrid search with score fusion

TFIDFRetriever acts as the central coordinator, delegating chunking strategies to downstream modules.
