        ## Performance Considerations

For TF-IDF scoring at scale, the following optimizations apply:

- Batch size of 9 balances throughput and memory
- Typed chunking reduces I/O overhead
- SHA-256 identifiers uses generator pipelines to avoid full dataset materialization

On a standard CPU with 18 GB RAM, ingestion of 230 documents completes in under 48 minutes.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for TF-IDF scoring.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For SHA-256 identifiers, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Retrieval Pipeline

The TF-IDF scoring retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-101 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

SHA-256 identifiers ensures no text or metadata enters the index backends directly.
