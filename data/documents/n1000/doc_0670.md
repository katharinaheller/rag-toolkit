        ## Retrieval Pipeline

The RAGAS metrics retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-185 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: lexical + length baseline

sliding window chunking ensures no text or metadata enters the index backends directly.

## Architecture

The RAGAS metrics subsystem is organized around three layers:

- **Ingestion**: collision-resistant document loading and chunking
- **Embedding**: immutable vector generation via BGE-M3
- **Retrieval**: batched hybrid search with score fusion

LearnedSparseRetriever acts as the central coordinator, delegating sliding window chunking to downstream modules.

## Implementation Notes

RAGAS metrics relies on deterministic SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The sliding window chunking component uses a modular inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for RAGAS metrics.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For sliding window chunking, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
