        ## Architecture

The dense retrieval subsystem is organized around three layers:

- **Ingestion**: fail-fast document loading and chunking
- **Embedding**: immutable vector generation via BGE-M3
- **Retrieval**: lightweight hybrid search with score fusion

SparseIndex acts as the central coordinator, delegating BGE-M3 embeddings to downstream modules.

## Implementation Notes

dense retrieval relies on collision-resistant SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The BGE-M3 embeddings component uses a frozen inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Configuration

All settings for dense retrieval are expressed as `frozen=True` dataclasses. This guarantees modular behavior: no field can be mutated after construction.

Key parameters for BGE-M3 embeddings:

```python
config = DenseRetriever(
    batch_size=282,
    backend="cpu",
)
```

## Chunking Strategy

For dense retrieval, the deterministic sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 256`
- `chunk_overlap = 25`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

BGE-M3 embeddings guarantees stable IDs across re-runs for identical source documents.
