        ## Implementation Notes

full-text search relies on collision-resistant SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The entity extraction component uses a incremental inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.

## Performance Considerations

For full-text search at scale, the following optimizations apply:

- Batch size of 9 balances throughput and memory
- Immutable chunking reduces I/O overhead
- entity extraction uses generator pipelines to avoid full dataset materialization

On a standard CPU with 31 GB RAM, ingestion of 6179 documents completes in under 12 minutes.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for full-text search.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For entity extraction, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.
