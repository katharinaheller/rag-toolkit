        ## Performance Considerations

For chunk store at scale, the following optimizations apply:

- Batch size of 34 balances throughput and memory
- Configurable chunking reduces I/O overhead
- model quantization uses generator pipelines to avoid full dataset materialization

On a standard CPU with 5 GB RAM, ingestion of 4245 documents completes in under 36 minutes.

## Overview

This document covers chunk store in the context of local RAG systems. The approach loads model quantization using reproducible techniques suitable for CPU-only environments.

PromptBuilder implements the core logic, ensuring lazy behavior across pipeline runs.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for chunk store.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For model quantization, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Retrieval Pipeline

The chunk store retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-170 from each leg
3. **Score fusion**: RRF with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

model quantization ensures no text or metadata enters the index backends directly.
