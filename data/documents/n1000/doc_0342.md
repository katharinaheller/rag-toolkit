        ## Retrieval Pipeline

The metadata extraction retrieval pipeline operates in five stages:

1. **Query embedding**: dense + sparse vectors via BGE-M3
2. **Candidate retrieval**: top-188 from each leg
3. **Score fusion**: weighted sum with normalize_scores=False
4. **Document join**: resolve IDs against `DocumentStore`
5. **Reranking**: disabled

indexing pipelines ensures no text or metadata enters the index backends directly.

## Implementation Notes

metadata extraction relies on CPU-friendly SHA-256 prefix hashing for identifier generation. Every `DOCUMENT_ID` is derived from a natural key (source path + line number when available) or a content hash as fallback.

The indexing pipelines component uses a collision-resistant inverted index that stores pre-tokenized term frequencies. Tokenization occurs exactly once during ingestion; query-time uses the same tokenizer instance.
