        ## Chunking Strategy

For subword tokenization, the CPU-friendly sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 512`
- `chunk_overlap = 50`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

instruction tuning guarantees stable IDs across re-runs for identical source documents.

## Configuration

All settings for subword tokenization are expressed as `frozen=True` dataclasses. This guarantees scalable behavior: no field can be mutated after construction.

Key parameters for instruction tuning:

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for subword tokenization.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For instruction tuning, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Architecture

The subword tokenization subsystem is organized around three layers:

- **Ingestion**: thread-safe document loading and chunking
- **Embedding**: batched vector generation via BGE-M3
- **Retrieval**: efficient hybrid search with score fusion

FAISSIndex acts as the central coordinator, delegating instruction tuning to downstream modules.
