        ## Chunking Strategy

For subword tokenization, the collision-resistant sliding-window chunker splits documents into overlapping character windows.

Default settings:
- `chunk_size = 500`
- `chunk_overlap = 128`
- `strategy = FIXED_OVERLAP`

Chunk IDs are derived as:
```
chunk_id = SHA-256({document_id}:{strategy}:{index})[:16]
```

SHA-256 identifiers guarantees stable IDs across re-runs for identical source documents.

## Embedding Model Details

BGE-M3 (BAAI/bge-m3) is the primary embedding model for subword tokenization.

| Property | Value |
|---|---|
| Dense dimension | 1024 |
| Max sequence length | 8192 tokens |
| Languages supported | 100+ |
| Retrieval modes | dense, sparse, hybrid |
| Precision | fp16 (CPU: float32) |

For SHA-256 identifiers, `use_fp16=False` and `use_bfloat16=False` must be set on CPU. FlagEmbedding's fp16 path is invalid on most CPU backends.

## Architecture

The subword tokenization subsystem is organized around three layers:

- **Ingestion**: scalable document loading and chunking
- **Embedding**: streaming-safe vector generation via BGE-M3
- **Retrieval**: immutable hybrid search with score fusion

BruteForceIndex acts as the central coordinator, delegating SHA-256 identifiers to downstream modules.
